# OneShelf PRD

## 1. 文档信息

- 项目名称：OneShelf
- 文档版本：V0.1
- 文档状态：已确认，可作为 MVP 设计与开发输入
- 首发平台：Android
- 开发框架：Flutter

## 2. 产品概述

OneShelf 是一个面向 Android 手机的本地媒体库应用，专注于读取用户已经整理好的本地媒体目录及元数据，为用户提供轻量、隐私友好、触屏友好的海报墙浏览与播放体验。

它不是一个完整媒体中心，也不是电视优先的大而全方案。OneShelf 的目标是围绕手机场景，解决本地媒体在 Android 上“难浏览、难管理、难顺畅播放”的问题。

一句话定义：

> OneShelf 是一个为 Android 设计的本地媒体海报墙与播放器，专注于读取本地元数据、浏览媒体库和流畅播放。

## 3. 项目背景与动机

当前已有方案无法很好满足目标用户在 Android 手机上的本地媒体使用场景。

### 3.1 Kodi 的问题

- 能读取本地元数据、展示海报墙、管理媒体库
- 但 UI 明显偏电视和横屏场景，手机触屏体验较差
- 在 Android 上对 SD 卡和本地目录访问的使用体验不够轻量

### 3.2 Nova 的问题

- 手机端体验相对更接近目标场景
- 但对本地元数据读取和展示不够稳定
- 容易受到新版 Android 存储权限限制影响

### 3.3 用户当前已有基础

用户已经通过 JavSP 等工具整理好本地媒体目录，目录中通常包含：

- 视频文件
- `.nfo`
- `poster.jpg`
- `fanart.jpg`
- 其他关联图片或信息文件

因此 OneShelf 的核心价值不是重新整理媒体，而是稳定读取、组织展示和流畅播放。

## 4. 产品目标

OneShelf 的目标不是做一个全功能 Kodi 替代品，而是做一个：

- 本地优先
- 手机优先
- 海报墙优先
- 元数据优先
- 播放体验优先
- 隐私优先

的轻量媒体库应用。

### 4.1 V1 产品目标

- 支持添加本地目录与 SD 卡目录作为媒体源
- 扫描目录并建立本地媒体索引
- 读取本地元数据与封面资源
- 提供适合手机浏览的海报墙
- 提供本地评分记录能力
- 支持详情页查看媒体信息
- 支持内置播放器播放
- 支持调用外部播放器
- 在无网络环境下完整可用

## 5. 目标用户与使用场景

### 5.1 目标用户

- 有本地影片收藏的 Android 手机用户
- 已经通过 JavSP 或类似工具整理过媒体目录的用户
- 重视隐私，不希望依赖云端媒体服务的用户
- 主要在手机上浏览和播放本地影片的用户

### 5.2 当前核心场景

用户的影片主要保存在：

- Android 手机本地存储
- TF / SD 卡

用户已经有统一或半统一的目录结构，希望在手机上：

- 浏览海报墙
- 查看本地元数据
- 快速找到目标影片
- 直接播放或交给外部播放器播放

### 5.3 后续可能扩展场景

以下场景是未来扩展方向，不属于 V1：

- 便携 NAS
- 外接 SSD 存储
- 局域网 SMB 共享目录
- WebDAV 共享目录

## 6. 目录结构与数据来源假设

当前用户典型目录结构示例：

```text
Sorted/
  演员名/
    [番号]/
      视频文件
      NFO
      poster.jpg
      fanart.jpg
```

该结构是重点兼容对象，但不是唯一合法结构。OneShelf 应以“扫描本地文件夹 + 读取同目录资源”为基础能力，兼容轻微差异化的实际目录与命名习惯。

### 6.1 V1 数据来源原则

- 本地文件系统是唯一权威数据源
- 本地元数据为展示主依据
- 不依赖在线刮削或云端补全
- 不要求用户额外维护服务端或数据库

## 8. 核心用户故事

### 8.1 媒体导入

作为用户，我希望添加手机本地或 SD 卡中的目录作为媒体源，以便应用自动识别我的影片。

### 8.2 海报墙浏览

作为用户，我希望在手机上以海报墙形式浏览本地影片，而不是只能通过文件列表找视频。

### 8.3 本地元数据查看

作为用户，我希望直接看到本地已有的封面、简介、演员、番号等信息，而不需要联网获取。

### 8.4 快速播放

作为用户，我希望从详情页一键内置播放，也可以选择交给习惯使用的外部播放器。

### 8.5 断网可用

作为用户，我希望即使完全离线，也能正常浏览媒体库与播放影片。

### 8.6 故障可恢复

作为用户，我希望当目录权限失效、封面丢失或文件被移动时，应用能给出明确提示并允许我恢复，而不是静默出错。

## 10. 交互与移动端设计原则

OneShelf 的交互设计必须服从 Android 手机使用场景，而不是照搬桌面或电视端模式。

### 10.1 设计原则

- 竖屏浏览优先，横屏播放优先
- 单手操作优先，主要 CTA 落在拇指热区
- 高频交互入口优先放在底部安全区域附近，而不是顶部边缘
- 所有关键动作都有显式入口，不依赖手势作为唯一方式
- 触控目标尺寸符合移动端可点击标准
- 长列表和海报墙优先保证流畅而非复杂特效
- 加载、扫描、错误、重试都必须有明确反馈

### 10.2 UI 方向

- 媒体库首页以海报墙为主，不走复杂菜单树
- 减少层级跳转，让“选中影片并播放”路径尽量短
- 媒体库页顶部尽量保持轻量，不放置高频操作按钮
- 搜索、设置、筛选等高频辅助动作优先通过底部工具栏承载
- 详情页信息组织以“先看封面和播放，再看元数据”为主
- 播放过程避免频繁误触，控制项要适配横屏与单手回退

## 11. 功能需求细化

### 11.1 媒体源接入

- 用户可通过系统目录选择器选择一个目录
- 应用保存目录访问权限
- 应用应区分“已接入”“权限失效”“扫描中”等状态

### 11.2 扫描规则

- 扫描时识别视频文件及同目录元数据文件
- 每个媒体项以文件夹或视频文件为基础建立索引
- 当目录结构不完全符合示例时，仍尽可能识别同目录的媒体资源
- 当 NFO 缺失时，以文件名和基础文件信息兜底
- 扫描时应识别标准附属视频命名，并避免将预告、花絮、采样片等附属视频误建为独立媒体
- 在单影片目录下，默认只建立一个主媒体项；被识别为附属内容的视频不进入主海报墙

### 11.3 媒体项标识规则

- V1 采用基于本地路径的确定性媒体 ID 方案，而不是随机生成 ID
- 当目录符合“单影片单文件夹”模式时，优先以媒体源内相对文件夹路径生成 `media_id`
- 当无法稳定识别媒体文件夹时，回退为以媒体源内相对主视频路径生成 `media_id`
- 评分、播放进度、最近观看等本地状态都绑定到 `media_id`
- V1 不保证文件夹重命名或跨目录移动后仍保留本地状态；该能力留待后续版本增强

### 11.4 附属视频识别规则

- V1 参考 Kodi 与 Jellyfin 的本地媒体命名思路，优先按显式命名规则识别附属视频
- 识别对象包括但不限于：`Extras` 目录、`trailers`、`featurettes`、`interviews`、`deleted scenes`、`behind the scenes`、`clips`、`samples`、`other`
- 同目录文件名若带有 `-trailer`、`.trailer`、`_trailer`、`-sample`、`.sample`、`-featurette`、`-interview`、`-behindthescenes` 等标准后缀，应视为附属视频
- 附属视频在 V1 只参与识别与排除，不作为独立主媒体展示；后续版本可再考虑在详情页中展示

### 11.5 元数据展示规则

- 优先使用本地 `.nfo` 中的标题、演员、简介等字段
- 优先使用 `poster.jpg` 作为海报
- 优先使用 `fanart.jpg` 作为详情页背景图
- 若图片损坏或不存在，显示默认占位图

### 11.6 搜索与评分规则

- V1 仅支持按文件名进行基础搜索
- 番号、演员等元数据搜索后置到后续版本
- 用户可在详情页设置 0-10 分的本地评分
- 本地评分仅保存在应用本地数据库中，不写回 NFO

### 11.7 播放规则

- 内置播放器为默认推荐方案
- 用户可手动选择使用外部播放器打开
- 应记录播放进度，支持继续观看
- 对不可播放文件要给出明确错误提示
- 媒体库首页顶部应在存在进度记录时展示一行最近观看
- 播放页在显式播放控件之外，可提供快捷手势：
  - 双击中间：暂停/继续
  - 双击左侧：后退
  - 双击右侧：前进
  - 长按右侧：临时倍速播放
- 所有快捷手势都不是唯一入口，播放、暂停、拖动与返回仍需有显式控件
- 前进/后退时长与长按倍速值应支持用户在设置页中修改
- 默认值建议为：
  - 前进：10 秒
  - 后退：10 秒
  - 长按右侧倍速：2.0x

### 11.8 排序与浏览

- 支持最近加入排序
- 支持基础字母或导入顺序排序
- 支持按文件夹浏览
- 按演员浏览可作为增强能力在首版后半程或 V1.1 完善

## 12. 非功能需求

### 12.1 性能

- 在 1000+ 媒体条目场景下保持海报墙可流畅滚动
- 封面加载应异步处理，不阻塞主界面
- 扫描应后台执行，不阻塞浏览

### 12.2 隐私

- 默认不上传媒体信息
- 默认不依赖第三方云服务
- 默认不需要账号系统

### 12.3 离线能力

- 无网络时，核心功能应完整可用
- 媒体浏览、元数据展示、播放功能均不依赖网络

### 12.4 可靠性

- 权限失效时应提示重新授权
- 文件丢失时应标记不可用状态或在重扫后移除
- 扫描失败时应有错误提示与重试入口

### 12.5 可扩展性

- 媒体源访问层需与媒体索引层解耦
- 未来可扩展至 SMB / WebDAV 等网络目录
- V1 数据模型设计应为未来多媒体源并存留出空间

## 15. 风险与注意事项

- Android 新版存储权限机制可能影响目录稳定访问，需在技术设计阶段优先处理
- 海量封面加载和本地扫描性能需要在架构上提前控制
- 不同用户目录结构存在差异，扫描规则不能只绑定单一 JavSP 命名格式
- 外部播放器联动在不同 ROM 和设备上的行为可能存在兼容性差异

## 16. 决策记录

### 决策 1：Android 优先且使用 Flutter

- 备选方案：原生 Android、跨平台方案
- 选择原因：当前目标场景明确为 Android 手机，Flutter 适合快速构建高质量移动 UI，并保留后续扩展空间

### 决策 2：V1 只支持本地目录和 SD 卡

- 备选方案：首版同时支持 SMB / WebDAV
- 选择原因：本地目录与 SD 卡是当前最高频且最明确的场景，先把核心体验打磨完整

### 决策 3：元数据只读，不提供编辑

- 备选方案：在应用内支持 NFO、封面编辑
- 选择原因：用户已有成熟整理链路，OneShelf 的价值在于读取、展示和播放，而不是再次承担整理职责

### 决策 4：竖屏浏览优先，横屏播放优先

- 备选方案：统一横屏或强电视风格界面
- 选择原因：符合手机端单手浏览和沉浸播放的使用习惯

### 决策 5：不做在线刮削与云功能

- 备选方案：补充在线元数据能力
- 选择原因：保持本地优先、隐私友好和产品范围聚焦

### 决策 6：媒体项采用基于路径的确定性 ID

- 备选方案：随机 ID、以完整视频文件路径作为唯一锚点
- 选择原因：Kodi 与 Jellyfin 都是明显的本地路径驱动模型。OneShelf 采用“优先文件夹、回退主视频路径”的确定性 ID，更贴合用户现有单影片文件夹结构，也能在主视频文件名变化但目录未变时保留本地状态

### 决策 7：首页即媒体库海报墙

- 备选方案：独立首页 + 二级媒体库页
- 选择原因：手机场景下应缩短进入海报墙与播放的路径，最近观看只需作为媒体库顶部一行即可

### 决策 8：V1 搜索仅保留文件名搜索

- 备选方案：首版同时支持番号、演员、评分等多字段搜索
- 选择原因：文件名搜索实现简单、价值明确，能满足首版基础找片需求，复杂索引留待后续版本

### 决策 9：V1 增加本地评分

- 备选方案：不做评分，或写回 NFO
- 选择原因：本地评分是轻量但高价值的个人管理能力，且不依赖联网，也不要求修改原始元数据文件

## 17. 确认结论

以下范围已经确认：

- 同意 V1 只做本地目录与 SD 卡，不做 SMB / WebDAV
- 同意元数据只读，只展示封面与本地信息，不提供编辑
- 同意 UI 以手机竖屏浏览为主，播放页重点适配横屏

该 PRD 可作为下一阶段以下工作的输入：

- MVP 功能拆解
- 页面流程设计
- 数据模型设计
- Flutter 技术方案设计

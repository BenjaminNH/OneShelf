# OneShelf MVP 功能清单

## 1. 文档定位

本文档用于在已确认的 PRD 基础上，进一步定义 OneShelf 的 MVP 范围、优先级与最小验收标准。

目标不是罗列所有未来想法，而是明确：

- 什么是首版必须交付的能力
- 什么可以作为增强项放在首版后半程或 V1.1
- 什么明确不进入 MVP

本文档默认服务于 Android 首发版本，且遵循以下前提：

- 平台：Android 手机
- 技术：Flutter
- 数据来源：本地目录与 SD 卡
- 元数据策略：只读，不编辑
- 联网策略：离线可用，不依赖在线刮削

## 2. MVP 定义

OneShelf 的 MVP 应完成一个最短但完整的闭环：

1. 用户可以接入一个本地目录或 SD 卡目录。
2. 应用可以扫描该目录中的视频与本地元数据。
3. 用户可以在手机上以海报墙形式浏览这些媒体。
4. 用户可以查看详情并播放视频。
5. 用户在媒体文件、封面或权限出现异常时，能够得到可理解、可恢复的反馈。

如果没有形成这条完整链路，则不算 MVP 完成。

## 3. MVP 分层原则

### 3.1 P0：必须有

没有这些功能，OneShelf 无法完成“本地海报墙 + 播放器”的基本价值闭环。

### 3.2 P1：应该有

这些能力明显提升可用性和完成度，但不应阻塞首个可用版本上线。

### 3.3 P2：可以后置

这些能力有价值，但应在首版核心体验稳定后再考虑。

## 4. P0 必须有功能

### 4.1 媒体源接入

#### 功能说明

- 支持添加一个或多个本地目录
- 支持添加 SD 卡目录
- 支持展示媒体源当前状态

#### 最小验收标准

- 用户可以通过系统目录选择器成功选择目录
- 应用可以保存目录访问授权
- 应用重启后仍能识别已添加媒体源
- 当媒体源权限失效时，应用可以明确提示并引导重新授权

### 4.2 首次扫描与重扫

#### 功能说明

- 支持首次全量扫描
- 支持手动重新扫描
- 支持识别新增文件与失效文件

#### 最小验收标准

- 用户在添加媒体源后可触发扫描
- 扫描期间 UI 不会卡死
- 扫描结果可反映媒体总数
- 删除或移动文件后，重扫可更新结果
- 扫描失败时有可见错误信息
- 标准命名的预告、花絮、采样片等附属视频不会被误加入主海报墙

### 4.3 本地元数据读取

#### 功能说明

- 读取同目录 `.nfo`
- 读取 `poster.jpg`
- 读取 `fanart.jpg`
- 读取视频文件的基本属性

#### 最小验收标准

- 当同目录存在 `poster.jpg` 时，海报墙优先显示该图片
- 当同目录存在 `fanart.jpg` 时，详情页可显示背景图
- 当 `.nfo` 存在时，可读取标题、简介、演员、番号等核心字段
- 当元数据不完整时，应用使用文件名和默认占位图降级展示

### 4.3.1 媒体项标识

#### 功能说明

- 为每个媒体项生成确定性的本地 `media_id`
- 优先使用媒体源内相对文件夹路径生成媒体 ID
- 无法稳定识别文件夹时回退为主视频相对路径

#### 最小验收标准

- 同一媒体在重复扫描后 `media_id` 保持不变
- 用户的本地评分与播放进度绑定到 `media_id`
- 在“主视频文件名变化但媒体文件夹不变”的情况下，本地状态不会丢失

### 4.4 海报墙浏览

#### 功能说明

- 首屏即媒体库海报墙
- 顶部提供一行最近观看
- 以网格海报墙展示媒体库
- 支持进入详情页
- 支持基础排序

#### 最小验收标准

- 应用启动后默认进入媒体库海报墙，而不是独立首页
- 当存在播放进度记录时，媒体库顶部可展示一行最近观看
- 海报墙可在手机竖屏正常浏览
- 封面缺失时不会导致布局错乱
- 用户点击海报可以进入详情页
- 支持至少一种默认排序方式，如最近加入

### 4.5 详情页

#### 功能说明

- 展示海报、背景图、标题、番号、演员、简介
- 展示基础文件信息
- 提供播放入口

#### 最小验收标准

- 用户从海报墙进入详情页后可看到核心元数据
- 信息缺失时使用合理占位而不是空白异常
- 详情页可一键进入播放
- 详情页可提供“使用外部播放器打开”

### 4.6 内置播放

#### 功能说明

- 内置播放器播放本地视频
- 支持横屏播放
- 提供基础播放控制
- 提供播放器快捷手势

#### 最小验收标准

- 用户可以在应用内播放已识别的视频文件
- 播放页支持播放、暂停、进度拖动、返回
- 播放页支持以下快捷手势：
  - 双击中间暂停/继续
  - 双击左侧后退
  - 双击右侧前进
  - 长按右侧临时倍速播放
- 前进/后退时长与长按倍速值可在设置中修改
- 默认值为后退 10 秒、前进 10 秒、长按右侧 2.0x
- 横屏播放可用
- 不可播放文件会出现明确错误提示

### 4.7 外部播放器调用

#### 功能说明

- 从详情页或播放入口调用系统外部播放器

#### 最小验收标准

- 用户可以成功使用外部播放器打开视频
- 当设备没有可用外部播放器时，有清晰提示

### 4.8 基础搜索

#### 功能说明

- 仅支持按文件名进行基础搜索

#### 最小验收标准

- 用户输入关键词后能返回匹配结果
- 无结果时有空状态提示
- 搜索结果可直接进入详情页

### 4.9 本地评价打分

#### 功能说明

- 支持用户为媒体记录本地评分
- 评分范围为 0-10 分
- 评分仅保存在本地，不写回 NFO

#### 最小验收标准

- 用户可在详情页为当前媒体设置或修改评分
- 已评分媒体再次打开时可正确显示此前评分
- 未评分媒体应显示为空状态而不是默认伪造分数
- 评分数据在应用重启后仍可保留

### 4.10 播放进度记录

#### 功能说明

- 记录基础播放进度
- 支持继续观看

#### 最小验收标准

- 用户退出播放后可保留当前进度
- 再次进入详情页时可看到继续播放入口
- 文件失效时不显示错误进度入口

### 4.11 基础设置

#### 功能说明

- 管理媒体源
- 手动重扫
- 基础播放偏好
- 播放器手势默认值设置

#### 最小验收标准

- 用户可查看已添加媒体源列表
- 用户可触发手动扫描
- 用户可选择默认播放方式或查看当前播放方式
- 用户可设置播放器默认前进秒数、后退秒数与长按倍速值

### 4.12 空状态与错误恢复

#### 功能说明

- 处理首次无媒体
- 处理权限失效
- 处理文件缺失
- 处理封面缺失

#### 最小验收标准

- 首次打开无媒体时有明确引导
- 权限失效时可重新授权
- 文件不存在时有合理提示
- 缺图、缺 NFO 时应用仍可工作

## 5. P1 应该有功能

### 5.1 按演员浏览

- 基于本地元数据中的演员字段聚合

价值：

- 提升从“目录浏览”到“媒体库浏览”的完成度

### 5.2 按文件夹浏览

- 允许用户保留一定程度的原始目录感知

价值：

- 对本地媒体用户很实用，尤其在元数据不完整时

### 5.3 扫描进度详情

- 展示当前媒体源扫描中、已完成、失败数量等状态

价值：

- 提高可理解性，减少“是否卡住”的不确定感

### 5.4 封面缓存管理

- 基础缓存清理
- 缓存占用展示

价值：

- 减少长期使用后的存储压力

### 5.5 默认外部播放器偏好

- 记录用户偏好的外部打开方式

价值：

- 减少重复选择成本

## 6. P2 可以后置功能

### 6.1 更复杂的排序与筛选

- 多维排序
- 条件筛选
- 多标签组合浏览

### 6.2 进阶搜索

- 按番号搜索
- 按演员名搜索
- 按评分等字段筛选

### 6.3 播放历史页

- 完整历史记录
- 最近播放列表

### 6.4 更丰富的媒体详情字段

- 更多技术参数
- 更复杂的图片资源展示

### 6.5 更智能的增量扫描

- 更细粒度的目录监听
- 更强的文件变更检测

## 7. 明确不进入 MVP 的内容

以下内容即使有吸引力，也不应进入 MVP 排期：

- SMB 接入
- WebDAV 接入
- NAS 直连
- 在线刮削
- 在线元数据修复
- NFO 编辑
- 封面替换与编辑
- 账号系统
- 云同步
- 社区、推荐、评论
- 电视端 UI
- 平板专属大屏适配

## 8. 推荐上线切片

为了降低首版交付风险，建议将 MVP 再拆成两个开发切片。

### 8.1 Slice A：打通核心链路

- 添加媒体源
- 首次扫描
- 本地元数据读取
- 海报墙
- 详情页
- 内置播放

目标：

- 先证明 OneShelf 的核心闭环是成立的

### 8.2 Slice B：补齐可用性

- 外部播放器调用
- 搜索
- 本地评价打分
- 播放进度记录
- 设置页
- 权限失效处理
- 空状态与错误恢复

目标：

- 让首版达到可长期使用的基础标准

## 9. MVP 完成定义

满足以下条件时，可视为 OneShelf MVP 完成：

- 用户可以成功添加本地目录或 SD 卡目录
- 应用可以扫描并识别至少一批本地媒体
- 用户可以在海报墙中流畅浏览媒体
- 用户可以打开详情页并查看本地元数据
- 用户可以通过内置播放器播放视频
- 用户可以调用外部播放器
- 用户可以通过文件名搜索媒体
- 用户可以为媒体记录和修改本地评分
- 用户的基础播放进度可以被记录和恢复
- 应用对无媒体、无封面、无 NFO、权限失效、文件失效有可用反馈

## 10. 决策补充

### 决策 A：MVP 仍保留搜索

- 原因：虽然搜索不是最小播放链路的一部分，但文件名搜索成本低、价值明确，适合作为 MVP 的基础检索能力

### 决策 B：MVP 保留播放进度

- 原因：对于手机场景，继续观看是高频行为，缺失会显著削弱产品完成度

### 决策 C：进阶搜索后置到 P2

- 原因：番号、演员等字段搜索依赖元数据质量与索引设计，首版可以先用文件名搜索满足基础找片需求

### 决策 D：MVP 新增本地评价打分

- 原因：本地评分是轻量但高价值的个人管理能力，且不依赖联网、不要求写回元数据，适合纳入首版

### 决策 E：按演员浏览列为 P1

- 原因：它有明显价值，但依赖元数据质量，首版不应让其阻塞基础链路

### 决策 F：首页即媒体库

- 原因：手机场景下应直接进入海报墙，最近观看作为顶部一行即可，不需要独立首页

### 决策 G：媒体项采用确定性本地 ID

- 原因：评分和播放进度都需要稳定锚点。优先绑定媒体文件夹，能更好适配现有的单影片目录结构

## 11. 下一步建议

基于当前 PRD 与 MVP 清单，推荐下一步按以下顺序继续设计：

1. 页面流程与导航结构
2. 数据模型与扫描索引策略
3. Flutter 项目结构与技术选型
4. 首批页面低保真线框
